The headline decision

Load Amplitude via the Unified Script (CDN) and use GTM as the event orchestrator — not as the Amplitude loader. Mobile uses native SDKs.

⚡ The one thing to remember
The GTM Amplitude template cannot run Web Experiment — a product CCEP purchased. That single fact drives the entire web architecture. See Why this approach.

Why this approach

Every recommendation here is traceable to a product constraint, not a preference. Here's the reasoning you can put in front of the customer.

The deciding constraint: Web Experiment

Amplitude offers three ways to install on web, and they support different product sets. Because CCEP bought Web Experiment, the GTM template is ruled out as the primary loader.

ProductGTM TemplateUnified Script (CDN)Unified SDK (npm)
Analytics✓✓✓
Session Replay✓ checkbox✓✓
Guides & Surveys✓ not recommended✓ (+script)✓
Web Experiment (visual A/B)✗✓ only here✗
Feature Experiment (code flags)✗✗✓ (not purchased)
ℹ How to read the ✓
In the Unified Script (CDN) column, ✓ means included in the one script tag — no separate install. Analytics, Session Replay and Web Experiment all load from that single tag. The only exception is Guides & Surveys (✓ (+script)), which adds one more CDN tag (...engagement.js) purely for bundle size. So the whole web runtime is two tags in <head>.
⚠ Don't confuse the two "Experiment" products
Web Experiment = no-code visual A/B testing (what CCEP bought, only in the Unified Script). Feature Experiment = code-based feature flags (in the npm Unified SDK, not purchased). They are not interchangeable.

Why the Unified Script (CDN) wins for web

  • Only method that includes Web Experiment — non-negotiable given the contract.
  • One tag = Analytics + Session Replay + Web Experiment, with Amplitude-managed versioning (bug fixes/perf ship centrally, no CCEP release needed).
  • G&S is a single extra line via the engagement script.
  • Anti-flicker: hard-coded high in <head>, Web Experiment mutates the DOM before render — no content flash.

Why GTM stays — but changes role

We keep GTM (the team knows it, governance lives there), but repurpose it: instead of loading Amplitude, GTM reads the dataLayer and calls the already-initialised window.amplitude. One SDK instance, no double-counting. Full detail in GTM FAQ.

Why native SDKs for mobile

GTM's mobile "support" is Firebase-based and has no Amplitude template, and can't run Session Replay or G&S. Native Swift/Kotlin SDKs are the only path that delivers the purchased products on iOS/Android. See Mobile setup.

Cross-cutting decisions

One canonical identity

Use one user_id (the Salesforce / Cloud Craze customer/contact ID) identically on Web, iOS, Android so cross-platform user & account resolution works.

One EU project + country property

Phase 2 scales via a country event/user property in a single EU project — enables cross-country reporting while keeping one taxonomy. (Alternative: project-per-country for hard isolation.)

EU data residency

Set serverZone: "EU" everywhere and use EU endpoints. Confirm EU CDN/script endpoints during Sprint 0.

Test project first

Account/group data is not retroactive. Validate in a test project before production go-live.

Web setup

Unified Script for the runtime, GTM for event orchestration. Order in the <head> matters.

Step 1 — Head order (this is the important bit)

Amplitude loads before GTM so Web Experiment can suppress flicker. GTM's own snippet still goes high in head, just after Amplitude.

index.html — <head>
<!-- 1. Amplitude runtime: Analytics + Session Replay + Web Experiment -->
<script src="https://cdn.amplitude.com/script/API_KEY.js"></script>

<!-- 2. Guides & Surveys (separate script, synchronous, after Analytics) -->
<script src="https://cdn.amplitude.com/script/API_KEY.engagement.js"></script>

<script>
  window.amplitude.init("API_KEY", {
    serverZone: "EU",          // EU data residency
    autocapture: true,
    fetchRemoteConfig: true
  });
  window.amplitude.add(window.sessionReplay.plugin({ sampleRate: 1 }));
  window.amplitude.add(window.engagement.plugin());
</script>

<!-- 3. Google Tag Manager (after Amplitude) -->
<script>(function(w,d,s,l,i){ /* GTM-K8Q34SB3 loader */ })(...)</script>
⚠ Two rules from the docs
Don't set async=true on the Amplitude script (Analytics must load before engagement). Use the same API key for Analytics and G&S — mismatched keys corrupt data.

Step 2 — GTM as orchestrator (not loader)

Do not add the Amplitude GTM template — it would create a second SDK instance. Instead, GTM tags call the global instance. Example Custom HTML tag firing on a dataLayer event:

GTM · Custom HTML tag
<script>
  if (window.amplitude) {
    window.amplitude.track("Order Placed", {
      order_id:    {{DLV - order_id}},
      order_value: {{DLV - order_value}},
      outlet_id:  {{DLV - outlet_id}}
    });
  }
</script>
✓ Result
One Amplitude runtime on the page; GTM stays the governance layer for events; Web Experiment, Session Replay, and G&S all run off the same instance and identity.

Container in use for the demo

The demo environment uses container GTM-K8Q34SB3 ("myccep test"). Swap in CCEP's production container per market in Phase 2.

Mobile setup

Native SDKs only. GTM plays no role here. No Web Experiment on mobile (web-only product).

✋ Do not try to use GTM for mobile
GTM mobile = Firebase, has no Amplitude template, and can't run Session Replay or G&S. Full explanation in GTM FAQ.

iOS — Amplitude Swift SDK

AppDelegate.swift
import AmplitudeSwift

let amplitude = Amplitude(configuration: Configuration(
    apiKey: "API_KEY",
    serverZone: .EU
))
// Session Replay + Guides & Surveys as plugins
amplitude.add(plugin: SessionReplayPlugin(sampleRate: 1.0))
amplitude.add(plugin: EngagementPlugin())   // Guides & Surveys iOS SDK

Android — Amplitude Kotlin SDK

Application.kt
val amplitude = Amplitude(
    Configuration(
        apiKey = "API_KEY",
        context = applicationContext,
        serverZone = ServerZone.EU
    )
)
// Session Replay + Guides & Surveys as plugins
amplitude.add(SessionReplayPlugin(sampleRate = 1.0))
amplitude.add(EngagementPlugin())   // Guides & Surveys Android SDK
ℹ Exact plugin class/package names
Confirm the precise Session Replay and Guides & Surveys plugin/package names against the current SDK docs during Sprint 0 — the pattern (init SDK → add plugins) is stable, names evolve.
⚠ App-store release cadence is a scheduling dependency
Mobile changes ship on the app-store cycle. If releases are monthly, mobile instrumentation can't compress below that — factor into the roadmap.

Accounts & Salesforce

Group-level analytics for CCEP's B2B outlets/accounts, with attributes synced from Salesforce.

Model: user-level groups

MyCCEP is a B2B ordering portal — each login belongs to an outlet/account — so use user-level groups: set once via identify, persists for the user, attributes all their events to the account.

Web · assign account group
const id = new amplitude.Identify();
amplitude.setGroup("account", "SFDC_00123456");   // group value = Salesforce Account ID
amplitude.identify(id);

Salesforce sync for group properties

  • Group value = stable Salesforce Account ID (unique, typically numeric) — it's the sync key and must exist in Salesforce.
  • Send a human-readable Account Name group property so analysts can segment by it.
  • Salesforce integration runs a daily group-property sync (first sync the morning after activation); interval configurable.
  • Use the Group Identify API for properties not in Salesforce — both can run together.

Design limits

  • Up to 5 group types per project
  • Up to 1,000 account-level properties

Group value example

Type: account · Value: 00123456 · Property: Account Name = "Tesco Metro Kingston"

✋ Not retroactive — decide the model before go-live
Group associations and properties only apply to new data. There is no historical backfill. Lock the group model in Sprint 0 and validate in a test project.

GTM FAQ

The questions CCEP's team will ask about Google Tag Manager — with the answers.

"GTM says it supports iOS, Android, AMP, Server — can we do everything in GTM?"

No — and this is the most common GTM misconception. "GTM for Web" and "GTM for mobile" are different products:

GTM containerWhat it really isUse for Amplitude?
WebJavaScript in the browser — the Amplitude template targets this✓ event orchestration
iOS / AndroidFirebase-based; no Amplitude template; can't run Session Replay/G&S✗ use native SDKs
AMPAccelerated Mobile Pages✗ not relevant
Server (sGTM)Server-side tagging — could forward to Amplitude HTTP API◐ Phase 2+ option

So GTM supporting "all platforms" does not change the architecture: GTM (Web) orchestrates web events; mobile uses native SDKs.

"Why not just use the Amplitude GTM template?"

It can't run Web Experiment (purchased), and G&S via GTM isn't recommended. It also wraps its own Analytics instance, which would conflict with the Unified Script. We use GTM to call Amplitude, not to load it.

"Where does the Amplitude script go relative to the GTM snippet?"

Amplitude first (as high in <head> as possible, for Web Experiment anti-flicker), then the GTM snippet, then the GTM noscript after <body>. See Web setup.

"Can GTM inject the Amplitude script for us?"

Technically yes, but not recommended for Web Experiment — GTM often fires too late to prevent flicker. Hard-code the Amplitude runtime in <head>; keep GTM for events.
